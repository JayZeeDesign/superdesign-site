AI Agent Evals: A Practical Setup for Early-Stage Teams

Updated 27 May 2026· ~9 min read· For founders shipping agents

By the Superdesign team

TL;DR

  • 30 hand-labeled cases is the threshold where evals start catching real regressions. You can build that in a day.
  • Score on three axes: task success (binary), trajectory quality (LLM judge), and tool correctness (argument-level diff).
  • You do not need a framework. A JSONL file, a runner script, and a CI hook are the entire rig.
  • Run on every change that touches the agent — prompt, tools, model, even cache settings.
  • The biggest mistake is grading the final answer only. The bugs live in the trajectory.

Definition

AI agent evals

A fixed, repeatable test suite that scores an AI agent's behaviour on real tasks. Unlike single-response LLM evals, an agent eval grades the full multi-step trajectory: did the agent finish the task, did it pick sensible tools, and were the tool calls correct.

Most early-stage teams shipping AI agents in 2026 still do not have evals. They have vibes. They run the demo three times, decide it works, and merge. Two weeks later the agent regresses silently on a category of inputs they have never typed in person, and a Slack thread starts with "weird, this used to work."

This post is the eval setup we wish someone had handed us in week one. It is the cheapest 90% solution: thirty cases, three axes, twenty-four hours of focused work. It is deliberately not a research-grade harness — it is the founder-grade thing you can ship before lunch on a Wednesday.

If you have not read the evals section of our AI agent pillar guide, skim it first. This piece picks up where that one stops and walks you through the actual build.

Why agent evals are different from LLM evals

An LLM eval grades one input → one output. Did the model summarise the document well? Did it classify the email correctly? You can do this with a CSV and a spreadsheet column.

An agent does not produce one output. It produces a trajectory — a sequence of model calls, tool invocations, and tool results, ending in some outcome. Three things can be wrong about a trajectory:

  1. The agent never finished. (Task failure.)
  2. The agent finished but the path was bizarre. (Trajectory failure — five tool calls when one would do, or the agent ran in a circle for ten turns before stumbling onto the answer.)
  3. The agent finished with a sane path, but one tool call had the wrong arguments. (Tool failure.)

Grading only the final answer hides the second and third categories. They show up later as cost spikes, latency regressions, and the kind of "the agent does something weird sometimes" bug reports that are impossible to reproduce. Grade the trajectory.

Step 1 — Write 30 cases

Spend an afternoon pulling 30 real tasks. Real is the operative word. Synthetic cases generated by another LLM will look comprehensive and miss exactly the bugs that bite you. Eval data lives in the same places product feedback lives:

  • Your own usage logs. The first 100 traces from your private alpha are a goldmine — pick the 10 most interesting.
  • Bug reports. Every "the agent did X" complaint is a case waiting to be written.
  • Designer / non-engineer clicks. Hand the product to the least technical person on your team and watch what they try.
  • Adversarial inputs. What does the agent do when the user types nothing? Asks for something out of scope? Sends a five-page wall of text?

For each case, write down four things: the user input, the expected final answer (or a regex / property that has to hold), the expected tools that should be called (a set, not a strict sequence), and the shape of the trajectory ("should call search at least once," "should never call delete_user").

Store this as a JSONL file in your repo. One case per line. The file is the asset; everything else is a script.

{ "id": "case-001", "input": "Find the most recent invoice for ACME and email it to billing@acme.com",
  "expected": { "final_contains": ["sent to billing@acme.com"],
                "required_tools": ["search_invoices", "send_email"],
                "forbidden_tools": ["delete_invoice"] } }

Step 2 — Score on three axes

For each case, compute three scores. Keep them separate; collapsing them into a single number loses signal.

Task success — binary, rule-based

Did the agent produce the right final outcome? This is where you use cheap rule-based grading: regex on the final response, a property check on the database, an assertion that an email was sent. Aim for this to be code, not an LLM. Code is free, deterministic, and trustworthy.

Trajectory quality — LLM judge

Was the path sensible? Did the agent thrash, loop, or reach the answer in a straight line? Use a strong model (Claude Sonnet, GPT-4-class) as a judge with a tight rubric and 3-5 calibrated examples. Score 1–5 or pass/fail; do not invent a 100-point scale you cannot anchor.

Calibrate the judge against 30–50 human-labeled cases before you trust its scores. Inter-rater agreement between you and the judge needs to be roughly κ ≥ 0.6 (substantial agreement) before the judge becomes a useful signal.

Tool correctness — argument diff

For each expected tool call, diff the actual arguments against an expected schema or value. This catches the silent regressions: the model that used to pass { user_id: "u_123" } and now passes { userId: "u_123" } after you changed the system prompt. This is also code — write it once, reuse it forever.

Why three axes beat one number. A single accuracy metric drops 5% and you have no idea why. Three axes drop 5% and the dashboard tells you "task success steady, trajectory quality dropped 12%" — your agent now solves the same tasks but takes twice as long. You would have caught that exactly nowhere else.

Step 3 — Build the runner (one script)

The runner is a script that loads the JSONL, runs each case against your agent in parallel, captures the trace, scores it on the three axes, and writes results back to a results JSONL. That is the whole thing. ~200 lines of Python or TypeScript. No new dependency.

Three engineering tips that pay back later:

  • Persist full traces. Save the conversation, every tool call, every tool result. When a case fails, you want the trace, not a summary.
  • Stable seeds where possible. Set temperature: 0 for the eval runs. The signal you want is "did my change move the score," not "did a sampling roll change it."
  • Concurrency cap. Run 5–10 cases in parallel. Rate-limit budgets are easy to blow on a 60-case suite. Provider 429s are slower than just being patient.

Wire the runner into CI. Every PR that touches the agent runs the suite. Block merges on task-success regressions of more than one case. Flag (don't block) regressions on trajectory and tool-correctness — those need human eyes, not a red square in GitHub.

Step 4 — Care for the dataset like product code

Eval cases rot. They go stale when your product changes, when the model gets smarter, when your tool surface evolves. Treat the dataset like code: PRs to add cases, PRs to retire cases, a CODEOWNER who reviews changes. The set should grow by 1–3 cases a week from real bugs and shrink by 0–1 a week from retired surface area.

Two operational habits to build in week one:

  • "Every bug becomes a case." The PR that fixes a regression must include a new case that would have caught it. No exceptions. This is the only way the set stays representative.
  • "Eval scores live in the PR description." A PR that touches the agent shows the three-axis score before / after. Reviewers read the diff and the score. This is the cultural move; the tooling is secondary.

Where evals meet memory and orchestration

Two adjacent areas leak into your eval design and trip people up.

First, memory. If your agent has cross-task memory, your eval cases need a defined memory state. Most teams either reset to empty before each case (clean but unrealistic) or pre-seed a fixture profile (more realistic, more work). Pick one and document it. We unpack the patterns in AI agent memory patterns — the storage choice changes how you fixture your evals.

Second, orchestration. Multi-step orchestration frameworks (LangGraph, CrewAI, Mastra) make traces denser and harder to grade. Make sure your trace format is normalised before the judge sees it — strip framework noise, keep tool calls, model calls, and outcomes. Our framework comparison in AI agent orchestration framework comparison notes which frameworks expose clean trace APIs and which require unwrapping.

Anti-patterns we keep seeing

  • Grading only the final answer. You will miss regressions in trajectory and tool calls. See above.
  • Using LLM judges without calibration. A poorly-prompted judge produces noisier scores than no eval at all, with a more confident dashboard. Calibrate or do not ship the judge.
  • Letting the eval set grow past 500 cases without pruning. Once your suite takes 30+ minutes, engineers stop running it locally. The set has the wrong shape.
  • "We have evals" meaning a Notion page of example prompts. A list of prompts is not an eval; you cannot run it on every PR.
  • Confusing eval pass rate with product quality. A 95% pass rate on the wrong cases is a worse signal than 70% on the right ones. The dataset is the asset.

When to graduate from this setup

The 30-case / three-axis rig holds well from prototype to roughly a few thousand users. You will know it is time to level up when:

  • Your eval suite stops surfacing bugs your real users still hit. (Coverage gap — add cases from production traces.)
  • You can no longer trace a score change to a code change. (Move evals onto a managed platform — Braintrust, LangSmith, Langfuse — with experiment tracking.)
  • You ship more than once a day. (Add an online-evals layer — a slim judge running on a sample of production traffic, surfacing live regressions.)

None of those graduations matter on day one. Ship the JSONL and the runner first. The rest follows from real pressure, not imagined future scale.

FAQ

What are AI agent evals?

A fixed, repeatable test suite that scores an AI agent's behavior on real tasks. Unlike a single-response LLM eval, an agent eval grades the full multi-step trajectory: did the agent finish, did it pick sensible tools, and were the tool calls correct.

How many eval cases do I need to start?

Thirty hand-labeled cases is enough to start catching regressions. The marginal value drops fast after ~100 unless your task space is genuinely large. Most early-stage teams ship to ten thousand users on a 60-case suite and benefit more from labeling new failure modes than from doubling the existing set.

Should I use an LLM judge or human reviewers?

Both, with a split. Score task success with code (binary, cheap). Use an LLM judge for trajectory quality and tool sanity; calibrate it against 30–50 human-labeled cases first. Reserve human review for cases where the judge and the rule-based check disagree.

Do AI agent evals need a special framework?

No. A JSONL file of test cases plus a Python or TypeScript runner script is a complete rig. Braintrust, LangSmith, Langfuse, and Promptfoo are nice to have for dashboards, but your first six months of evals can live in your repo with no extra dependencies.

When should I run agent evals?

On every change that touches the agent: prompt edits, tool changes, model swaps, even prompt-cache changes. They belong in CI. A 60-case suite on a Sonnet or GPT-4-class model typically completes in 3–6 minutes with concurrency.

How is grading the final answer different from grading the trajectory?

Final-answer grading misses three classes of bug: silent tool failures the agent recovered from, paths that doubled in length, and the model passing slightly-wrong arguments to a tool that still succeeded. Trajectory grading catches all three. Both matter — but trajectory is where most regressions live.
