LLM Application Architecture: A Founder's Guide to Building with LLMs

Updated 27 May 2026· ~14 min read· For early-stage AI founders

By the Superdesign team

TL;DR

  • An LLM application is a model plus the seven layers around it. The model is the smallest part. The layers are where products live or die.
  • Don't fine-tune first. Prompt → few-shot → retrieval → fine-tune, in that order. Most founders never get past step three.
  • Pick a model by running 30–100 of your tasks on each candidate. Vendor benchmarks are marketing.
  • Structured output, prompt caching, and aggressive context trimming buy you the biggest cost and quality wins. Everything else is a rounding error.
  • Latency is a UX problem before it's an infra problem — stream first, optimize second.

Definition

LLM application

A software system that uses one or more large language models as part of a product feature. The architecture is the set of components and data flows around the model: input handling, prompt assembly, retrieval, model call, output validation, observability, and cost or latency controls. The model is one component in the system, not the system itself.

The most expensive mistake we see early AI teams make is treating the LLM as the product instead of treating the system around the LLM as the product. A founder ships a thin wrapper around chat.completions.create, calls it an "AI app," and watches it fall apart the first time a user gives it a real input. The model wasn't the problem. The missing architecture was.

This guide is the architecture diagram we wish every YC-batch AI team had on their first whiteboard. It's what we've learned building Superdesign — an LLM-powered design agent — and watching dozens of pre-seed teams build their own LLM features. It is opinionated, written for founders who have shipped backends but haven't yet shipped an LLM feature that survived ten thousand users.

We'll go through the seven layers of an LLM application, how to choose a model without burning a month, when retrieval beats fine-tuning (and when it doesn't), how to get reliable structured output, and the specific tactics that bring cost and latency to acceptable numbers. Each section links to a deeper post in the build-with-LLMs cluster when you want to go further.

The seven layers of an LLM application

Strip away the framework branding and a working LLM application has the same seven layers. If you can name them in your system, you can debug it. If you can't, something is hiding from you.

  1. Input layer — where user requests, files, screenshots, and events come in. Includes auth, rate limits, and basic input validation.
  2. Context assembly — what goes into the prompt. System prompt, user message, retrieved chunks, prior turns, examples. This is your product.
  3. Retrieval — pulling relevant facts from your corpus when the answer depends on them. Search, ranking, deduplication, recency biasing.
  4. Model call — the actual LLM invocation, including model choice, temperature, max tokens, tool definitions, and structured-output configuration.
  5. Output validation — schema-checking the response, retrying on failure, and quarantining responses that look unsafe or wrong.
  6. Observability and evals — traces, logs, replays, a fixed eval set, and a cost or latency dashboard your team actually looks at.
  7. Cost and latency controls — prompt caching, model routing, batch processing, streaming, and short-circuit rules.

None of these is exotic. All of them are missing from most demos. Build them in the order above; the layers later in the list quietly fail if the earlier ones are weak.

Founder-grade heuristic. If you cannot draw the seven layers of your LLM feature on a napkin in under two minutes, you don't have an architecture — you have a script. Scripts ship demo videos. Architectures ship products.

Choosing a model without burning a month

"Which model should I use?" eats more founder time than it should. The honest answer in 2026 is: it depends on your task, and the only way to know is to run your own evals. But here's a sober map of the realistic defaults.

Model familyStrongest forCost
Claude Sonnet 4.6 / Opus 4.7 Tool use, long context, writing quality, code, multi-turn agents. Mid-tier. Best price/quality if you use prompt caching.
GPT-5 / GPT-5 mini Raw reasoning, broad ecosystem, native structured output, batch API. Frontier-priced for GPT-5; mini is very cheap.
Gemini 2.5 Pro / Flash Multimodal, Google Cloud native stacks, very long context (1M+ tokens). Competitive; Flash is the cheapest serious option.
Open-weights (Llama 4, Qwen 3, Mistral Large) Self-hosting for compliance, fine-tuning, very high-volume cheap inference. Cheaper at scale; expensive in eng-hours.

None of these is wrong. They solve different problems. The mistake is picking one before you've run your own tasks against two of them. Build a fixed set of 30–100 representative inputs, run them through your top-two candidates, score on the dimensions that matter for your product. Two days of work. We go deeper in Choosing an LLM for production: a decision matrix.

The Claude-vs-GPT-for-agents question deserves its own pass, because the differences are real and product-shaped. Claude's tool-use accuracy and willingness to ask clarifying questions tend to make it the safer default for agents; GPT-5's reasoning chains and structured-output ergonomics make it strong for analysis tasks. See Claude vs GPT for agents for the head-to-head.

RAG vs. fine-tuning: when to use which

Founders pick RAG and fine-tuning in the wrong order. They reach for fine-tuning because it sounds sophisticated, struggle for two weeks, and end up with a model that's expensive to update and worse at general reasoning than the base. Then they bolt on retrieval to fix the gaps and ship a Frankenstein.

The boring, correct order is:

  1. Prompt engineering. Most quality gaps close with a better system prompt. Free, fast, debuggable.
  2. Few-shot examples. Include 3–10 representative input/output pairs in the prompt. Closes another large chunk of the gap. Cost: tokens.
  3. Retrieval (RAG). When the answer depends on facts the model can't memorize — documentation, ticket history, product catalogs, recent events.
  4. Fine-tuning. When you need a consistent style, format, or behavior that prompting cannot produce reliably even with good examples. Or when token economics make a smaller fine-tuned model cheaper at scale.

The two are not alternatives; they answer different questions. RAG handles "what should the model know?" Fine-tuning handles "how should the model behave?" The deeper decision tree — including hybrid setups where you fine-tune a small model for routing and use a frontier model for answering — lives in RAG vs. fine-tuning: when to use each.

If you do build RAG, the boring version is almost always right

The default RAG stack we recommend for a pre-seed team is unfashionable on purpose:

  • Storage: Postgres with the pgvector extension. One database, one set of credentials, one backup story.
  • Chunking: 500–1,000 token chunks with 10–15% overlap. Don't tune this before you have evals.
  • Retrieval: Hybrid — full-text search (BM25) plus vector similarity, then re-rank the top 50 with a cross-encoder if quality demands it.
  • Context budget: Inject the top 3–6 chunks. More retrieved context usually hurts; LLMs answer better with less, not more.

You do not need Pinecone, Weaviate, or a separate vector store for the first 6 months. You can need them later. Most teams do not get to later.

Structured output: the difference between a demo and a feature

Almost every LLM application eventually wants structured data — a JSON object, a function call, a typed result. The naive approach is to ask nicely and hope. That approach has a 70–95% success rate depending on the model and the schema, which is exactly bad enough to lose users without setting off alarms.

Three reliable patterns, in order of preference:

  1. Native structured output. OpenAI's response_format with a JSON Schema. Anthropic's tool-use as a typed return channel. Gemini's response schemas. These give you 99%+ schema conformance because they constrain decoding at the token level.
  2. Constrained-decoding libraries. Outlines, Instructor, or LMFE wrap the model to enforce a Pydantic / Zod schema even when the provider doesn't. Useful for self-hosted models.
  3. Prompt-then-validate-then-retry. Ask for JSON, parse with a schema library, retry once with the validation error injected into the next prompt. Works as a fallback. Don't make it your primary strategy.

Whatever you use, treat the parsed output as untrusted input. Validate every field. Reject nulls where you didn't allow them. Cap string lengths. The model will surprise you, and so will the user trying to break your model. Worked examples and pitfall cases live in Structured output from LLMs: patterns and pitfalls.

Founder-grade heuristic. If your "AI feature" depends on parsing free-form text from the model with a regex, you have a structured-output problem disguised as a parsing problem. Fix the upstream constraint, not the downstream regex.

Reducing LLM costs without sacrificing quality

The default cost curve of an LLM feature is exponential in the wrong direction. The fix is not "use a cheaper model" — it's a small set of architectural moves that compound.

  1. Prompt caching. Every major provider now offers it. Mark the stable prefix of your prompts — system prompt, tool definitions, retrieved boilerplate — as cacheable, and you cut repeated-prefix cost by 80–90%. This is the single biggest lever for most apps. Free to implement.
  2. Model routing. Send simple requests to a cheaper, faster model (Haiku, GPT-5 mini, Gemini Flash) and only escalate to a frontier model when a classifier says it's hard. Usually a 2–3× cost reduction on a typical request mix.
  3. Aggressive context trimming. LLMs answer better with less, not more. Trim retrieved chunks ruthlessly. Drop irrelevant turns from history. Use the long context window as a budget, not a target.
  4. Short-circuit rules. A regex, a SQL lookup, or a cached canonical answer can handle 20–40% of "AI" requests without ever hitting a model. Build the deterministic layer first; route everything else to the LLM.
  5. Batch API. Non-interactive workloads — overnight summarization, eval runs, content generation — go through the batch endpoint at roughly half price. Latency is hours, not seconds; for non-interactive paths nobody cares.

Cursor, Devin, Claude Code, and every other agent product you've heard of are doing all five of these on every request. The deep dive — including the specific cache-key shape and how to design prompts so the stable prefix is genuinely stable — is in How to reduce LLM costs in production.

Latency is a UX problem first

Founders chase latency by switching models. That's usually the third lever, not the first. The first lever is perception.

  • Stream tokens to the user. The model takes the same total time, but the user sees the first token in 300ms instead of 6s. For chat-shaped UX this is the entire game.
  • Show intermediate state. "Searching docs… Reading top result… Drafting answer…" — explicit progress events make a 10s response feel acceptable. Silent waits feel broken at 3s.
  • Parallelize independent work. Retrieval, classification, and the model call should run concurrently when they don't depend on each other. Most LLM apps serialize them by accident.
  • Cache and pre-compute. The hot 5% of queries can often be answered from a pre-computed cache with a freshness check. Sub-100ms responses for the common path; full pipeline for the long tail.

After those four, switch to a faster model on the latency-critical path. Haiku, GPT-5 mini, and Gemini Flash are 3–5× faster than their frontier siblings and good enough for routing, classification, and most extraction tasks. The full latency playbook — including speculative decoding, request hedging across providers, and KV-cache reuse for self-hosted models — is in LLM latency optimization for production.

Evals: how you stop shipping by vibes

The single biggest difference between LLM applications that quietly degrade and ones that quietly improve is whether the team has a real eval set. Hamel Husain's audit-style eval posts started this conversation in 2024, and two years later most pre-seed teams still ship every prompt change by vibes.

You do not need a research-grade harness. You need a fixed set of 30–100 real tasks with expected behaviors, run on every change. Score on three axes:

  1. Task success. Did the model solve it? Often easy to autograde — output matches expected, or contains a required substring, or passes a schema.
  2. Output quality. When success isn't binary (writing, summarization, design output), use an LLM judge with a rubric or human spot-checks on a sample.
  3. Cost and latency per task. Every prompt change that improves quality but doubles cost is a tradeoff, not an upgrade. Make the tradeoff visible.

Eval data comes from the same places product feedback comes from: real user transcripts, bug reports, edge cases your designer hit while clicking around. The first ten cases will feel arbitrary; that's fine. By case 40 you'll have a usable signal. The toolchain — Promptfoo, Braintrust, Langfuse, or a Postgres table — matters far less than the discipline of running the set on every change. LLM evals: a practical setup for early-stage teams walks through the toolchain.

Observability: agents and LLMs both die in production without it

An LLM application without traces is unfixable. You will get a Slack message that says "the AI answered something weird at 11pm, the user closed the tab." You need: which model, which prompt revision, the full input, the full output, the retrieved context, every tool call, cost and latency.

Minimum observability checklist for an LLM app:

  • Per-request trace with the assembled prompt, model parameters, full response, and parsed output. Store the payloads, not just metadata.
  • Cost and latency per request, surfaced in a dashboard your team sees daily.
  • A replay tool — given a trace ID, you can re-run the exact request against a new prompt or model.
  • Schema-validation failure rate, retry rate, and a per-model breakdown. When this number starts trending up, your structured-output prompts have drifted.
  • Tied-back conversion or success metric for the user task the LLM was supporting. Token counts are not a product metric.

LangSmith, Langfuse, Helicone, and Braintrust are reasonable defaults. So is a Postgres table with a JSON column and a small admin UI. The choice matters less than committing to actually open the traces every day for the first month.

From prototype to production: the five gates

This is the checklist we use internally before an LLM feature goes from "cool demo" to "real users can touch it." If your feature doesn't pass all five, don't ship. Ship the parts that pass.

  1. Narrow task. The feature does one thing well. "Summarize this PR into a release note" is narrow. "Be an engineering copilot" is not.
  2. Validated output. Every response goes through schema validation, length caps, and content filters before it reaches the user.
  3. Eval set passes. ≥90% task success on a 30+ case eval set, with no regressions on the last ten cases you added.
  4. Observability live. Traces, cost dashboard, error alerting wired up before the first real user touches it.
  5. Rollback plan. A feature flag to disable the LLM path and route to a deterministic fallback (a static answer, a form, a humans-only flow). Models get deprecated. Providers have outages. You will use this flag.

Common mistakes we keep seeing

From sitting next to enough early teams, the same five mistakes show up over and over:

  • Fine-tuning before prompting. Fine-tuning is the expensive third step, not the first. Prompt → few-shot → retrieval first; almost everyone stops there.
  • Stuffing the context window. "It has 1M tokens, let me put everything in." Models perform better with less. Trim ruthlessly.
  • Treating model output as trusted. No schema validation, no length cap, no content check. One day the model returns a 40K-character response and your UI catches fire.
  • Shipping without evals. Every prompt change after that is a coin flip dressed up as progress.
  • Picking a vector DB on day one. Postgres+pgvector covers you to thousands of users. Adopt a dedicated vector store when, not before, you can name what it gives you.

FAQ

What is LLM application architecture?

LLM application architecture is the set of components and data flows that turn a large language model into a reliable product feature. The minimum stack is input handling, prompt assembly, model call, output validation, and observability. Retrieval, caching, evals, and orchestration layer on as the product matures. The model is one component in a system, not the system itself.

When should I use RAG versus fine-tuning?

Use retrieval-augmented generation when the answer depends on facts that change or are too large for the prompt — docs, ticket history, product catalogs. Use fine-tuning when you need a consistent style, format, or behavior that prompting cannot reliably produce. Most early-stage teams need RAG, not fine-tuning. Try prompting and few-shot examples first; reach for fine-tuning only after evals show prompting plateaued.

Which LLM should I choose for production?

As of mid-2026 the realistic defaults are Claude Sonnet 4.6 or Opus 4.7 for tool use and long context; GPT-5 for raw reasoning and broad ecosystem support; Gemini 2.5 Pro for multimodal and Google-native stacks. Pick by running your own 30–100 task eval on each candidate. Vendor benchmarks are marketing; your eval set is the only signal that matters.

How do I get structured output from an LLM?

Use the native structured-output features of your model: OpenAI's response_format with a JSON Schema, Anthropic's tool-use as a typed return channel, or constrained-decoding libraries like Outlines and Instructor. Always validate the parsed output with a schema library (Pydantic, Zod) before using it. Treat every response as untrusted input until validation passes.

How do I reduce LLM costs in production?

Cache the stable prefix of your prompt — system prompt and tool definitions — using your provider's prompt-cache feature. Route simple requests to a cheaper model. Trim retrieved context aggressively. Short-circuit obvious cases with deterministic rules. Batch non-interactive workloads for roughly 50% off. Stacked, these typically cut effective cost 5–10× from a naive baseline.

How do I reduce LLM latency?

Stream tokens to the user instead of waiting for full responses. Cache stable prompt prefixes so repeated requests skip prefix processing. Run retrieval and the model call in parallel where dependencies allow. Pick a faster model class (Haiku, GPT-5 mini, Gemini Flash) for latency-critical paths. Pre-compute and cache common responses for the hot 5% of queries.

Do I need a vector database for an LLM application?

Only if you have a real corpus the model must search and your evals show retrieval improves answers. Most early-stage apps need a relational database with full-text search far more than they need a vector store. Postgres with pgvector covers both. Add a dedicated vector database when corpus size, query volume, or recall requirements make the boundary obvious.

How long does it take to build a production LLM feature?

A first version with a narrow task, validated structured output, a 30-case eval set, prompt caching, and observability is realistically 2–4 weeks for an experienced engineer. Most teams underestimate evals and observability and ship a prototype in a week that takes another month to harden. Plan for the second month.
