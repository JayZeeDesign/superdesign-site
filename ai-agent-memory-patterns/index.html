AI Agent Memory Patterns: Working, Episodic, and Semantic

Updated 27 May 2026· ~9 min read· For founders shipping agents

By the Superdesign team

TL;DR

  • Agent memory is three different problems — working, episodic, and semantic — that want three different storage layers.
  • Working memory lives in the context window. Trim aggressively; the model pays more attention to short contexts.
  • Episodic memory (user preferences, prior decisions) belongs in a relational table, not a vector index.
  • Semantic memory is the one a vector DB is for — and only if you have a real corpus.
  • The fastest agent-quality win in early teams is usually removing memory infrastructure, not adding more.

Definition

AI agent memory

Any persistence layer that lets an AI agent reference state beyond a single model call. Includes within-task scratchpads, across-task user history, and searchable knowledge corpora. The right storage depends on access pattern, not on what the tutorial said to use.

"How do I give my agent memory?" is the single most common question we get from early-stage founders, and almost every time the real answer is: you already have memory; you are just calling it the wrong thing. The conversation history is memory. The Postgres row with user preferences is memory. The vector index of past tickets is memory. They are different kinds of memory, and conflating them is what produces the bolt-everything-into-Pinecone architectures that quietly degrade agents in production.

This piece is the framework we use internally for picking the right memory layer per use case. It pairs with the memory section of our AI agent pillar guide, which gives the conceptual frame — here we go deeper on patterns and storage choices.

The three types of agent memory

From cognitive science, borrowed liberally and pragmatised for software engineers:

  • Working memory — the state of the current task. Lives in the context window. Ephemeral by definition.
  • Episodic memory — what happened across past tasks for this user. Preferences, history, "things we have already tried." Durable, user-scoped.
  • Semantic memory — knowledge about the world that is not user-specific. Product docs, internal wikis, past tickets. Durable, organisation-scoped, often large.

Each one has a different access pattern, a different update frequency, and a different correct storage choice. Picking the wrong store for one of them is a top-five reason agents misbehave under load.

Working memory: the context window is the database

Working memory is the simplest of the three because the storage is already given to you: the message array you pass to the model on each call. The hard part is not where to put it — it is how to keep it from rotting.

Three patterns that consistently work:

1. The scratchpad pattern

Give the agent a "scratchpad" tool — append-only notes the agent can read on subsequent turns. This sounds redundant with the conversation, but it works for a real reason: the model is much better at re-reading its own structured notes than at re-parsing past tool outputs. Anthropic and OpenAI both ship variations of this in their agent SDKs for the same reason.

2. The compression pattern

When the conversation grows past 50–70% of the model's context window, summarise older turns into a single "Summary so far" system message and drop the original turns. This is not free — summaries lose detail — but it is much better than the alternative, which is the model paying degrading attention to every turn as the window fills.

3. The structured-state pattern

For agents that work on long-running objects (a document, a design, a piece of code), do not put the object itself in the conversation. Put a handle to the object — an ID or a structured representation the agent can mutate via tool calls. The conversation stays tiny; the object lives in your database. Cursor and Claude Code both work this way.

Heuristic: if your agent's average context length grows linearly with turn count, you have a working-memory bug, not a "we need bigger context windows" problem. Models with a 1M-token context still pay shrinking attention as the window fills. Treat the budget as ~50–100K useful tokens regardless of advertised maximum.

Episodic memory: boring storage beats clever retrieval

Episodic memory is where most teams go wrong. The seductive idea is: "I will write every user interaction to a vector index, then retrieve relevant chunks at runtime." The reality: similarity search over user history returns plausible-sounding but topically wrong fragments, and the agent confidently builds a wrong answer on top.

Episodic memory is structured. Use structured storage. The pattern that works:

  • A user_facts table keyed by user ID. Columns for the things you actually need: timezone, current project, role, declared preferences. Updatable. Auditable. Queryable.
  • A user_history table of past tasks. One row per completed task, with input, outcome, and a short LLM-generated summary.
  • At task start, the agent runs a get_user_context(user_id) tool that pulls the facts row plus the 3–5 most recent history rows and returns them as a structured block.

That is it. No embeddings. No similarity search. No drift. The cost is one database query at task start; the benefit is deterministic, debuggable user context. When you eventually do need similarity over history — say, "have I done a task like this before?" — you can add an embedding column to user_history later. Premature retrieval is the new premature optimisation.

Semantic memory: the only place a vector DB earns its keep

Semantic memory is the corpus the agent searches at retrieval time. This is where vector databases actually fit. If you have:

  • Thousands of documents the agent might need to consult, and
  • No way to know in advance which ones,

...then embedding-based retrieval (or hybrid keyword + vector) is the right tool. Pinecone, Weaviate, Qdrant, pgvector, Turbopuffer — all fine. The choice between them matters far less than the choice to actually have a corpus worth searching.

Three patterns that we keep recommending:

  • Hybrid retrieval beats pure vector. Combine BM25 / keyword matching with semantic similarity, then rerank. Pure cosine similarity on small chunks misses exact-match queries (product names, error codes) that BM25 nails.
  • Chunking matters more than the vector DB. Chunk by semantic boundary (headings, paragraphs) not by token count. A 400-token chunk that ends mid-sentence costs you retrieval quality forever.
  • Always return the source. The agent should see the retrieved chunk and a citation. Sourceless retrieval makes hallucination indistinguishable from real answers.

Pick the right storage

Memory typeStorageAccess patternAnti-pattern
Working Context window + scratchpad tool Append, trim, summarise Letting context grow unbounded
Episodic Relational (Postgres/SQLite) Lookup by user_id; recent N rows Vector search over user history
Semantic Vector + keyword hybrid Query at task time, return chunks + cite Embedding a corpus you do not actually search

Anti-patterns that destroy agents quietly

  • "Memory" as a single bag. Stuffing chat history, preferences, and docs into one vector index produces results that are subtly wrong in subtly different ways. Split the memory.
  • Forgetting to TTL working memory. A scratchpad without expiry becomes a budget hog. Set a per-task limit; archive on completion.
  • Treating episodic memory as immutable. Users change. Preferences change. If the agent cannot update what it learned about a user last week, the system is brittle.
  • RAG without grounding. Returning chunks without citations forces the model to hallucinate fluently. Always pass source IDs through.
  • Embedding everything. Embeddings are not free — storage, latency, refresh costs. Embed corpora you actually retrieve from, nothing else.

Memory and evals are linked

Memory changes your eval contract. An eval case is no longer just a function of input — it is a function of input and memory state. There are two reasonable patterns:

  • Reset before each case. Working memory is empty; episodic memory points at a fresh fixture user. Clean signal, less realistic.
  • Pre-seeded fixtures. Each case names a fixture profile (e.g., "user_with_3_invoices") that the runner sets up first. More realistic; more state to maintain.

Whichever you pick, document it. The largest source of flaky agent evals we see is undeclared memory state — the case passes locally because your dev account has a quirk and fails in CI because the CI account does not.

If you have not built evals yet, start with our practical AI agent eval setup and pick the reset-before-each-case pattern. Move to fixtures only when reset becomes unrealistic.

When to evolve your memory architecture

Most teams over-engineer memory in week one and under-evolve it after launch. Two signs you need to invest more:

  • "The agent forgot what I told it yesterday." Episodic memory is missing or under-used. Add a user_facts table and a get_user_context tool before anything else.
  • "The agent answers from outdated knowledge." Semantic memory is missing or stale. Either add a real RAG layer with citations, or restrict the agent's scope to what it can do without world knowledge.

The pattern that destroys agents is the inverse: shipping a vector DB before you know what is in it, then debugging similarity-score weirdness for a week.

FAQ

What are the three types of AI agent memory?

Working memory holds state within a single task (the conversation, scratchpad, intermediate tool outputs). Episodic memory holds what happened across past tasks for one user (preferences, prior decisions, things you have already tried). Semantic memory is a corpus the agent can search at retrieval time (docs, tickets, product knowledge). Each wants different storage.

Do I need a vector database for an AI agent?

Only for semantic memory, and only if you actually have a corpus to search. Most early-stage agents have a working-memory problem and an episodic-memory problem, not a retrieval problem. Bolting a vector DB onto every memory need produces fuzzy similarity matches over fragments that should have been stored in a database with foreign keys.

How do I keep an AI agent from forgetting context within a task?

Use working memory deliberately. Put the conversation in the message history, put intermediate tool outputs in a scratchpad the agent can re-read, and summarize older turns when the context grows past 60% of the model's window. Aggressive trimming and a clean summary beat appending forever.

Where should I store user preferences for an AI agent?

In a relational table keyed by user_id, not a vector database. Preferences are structured, queryable, editable, and frequently updated — properties relational stores handle well and vector indexes handle badly. Inject a short summary into the system prompt or fetch on demand.

How does memory affect AI agent evaluation?

Memory changes the eval contract: results depend on prior state, not just current input. Either reset memory to a known state before every eval case, or pre-seed a fixture profile. Document the choice and keep it consistent — non-deterministic memory state is the largest source of flaky agent evals.

What's the difference between RAG and agent memory?

RAG is one technique for one type of memory — semantic. An agent's full memory stack includes working memory (context window state) and episodic memory (per-user history). Calling all of it "RAG" obscures the design choices that matter.
